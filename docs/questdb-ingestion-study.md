# What apitap Can Borrow from QuestDB — Engineering Brief

## 1. Bottom line

QuestDB is a database that owns its storage and runs as a persistent server. apitap is a stateless streaming transfer. Most of QuestDB's cleverest ideas assume owned, persisted, memory-mapped columnar files and do **not** transfer. But a handful of its *primitives* map cleanly onto apitap's known CPU-bound legs. In priority order (impact × cheapness):

1. **Move BQ gzip off the async worker thread (`spawn_blocking`).** Highest impact-per-line item on the board. The gzip compress currently runs *inline* inside `BqLoader::send()` (bigquery.rs:1141), blocking the tokio worker for the whole compress while the socket sits idle. This is the exact defect we already fixed for the Snowflake PUT (A1). Overlapping compress with read+upload is a medium-effort change to a leg we've explicitly flagged as gzip-CPU-bound on small cores. **Do this first.**

2. **Parquet DATE as `Vec<i32>` from the start.** A pure win with zero downside (verified: bqparquet.rs:87/511 stores `Vec<i64>` then allocates a throwaway `Vec<i32>` every row-group flush just to narrow). Kills one O(rows) alloc+copy per flush and halves the DATE buffer (8→4 B/cell). Small effort, real memory-ceiling relief.

3. **`Repetition::REQUIRED` + skip def-levels for NOT NULL columns.** Drop the entire `Vec<i16>` def-level array (one i16/row/column) and a per-cell push from the Parquet framing loop for wide all-NOT-NULL fact tables (verified: OPTIONAL hardcoded at bqparquet.rs:231, defs always pushed at :461/:467). Small effort, hits both write-encode CPU and the memory ceiling; nullability already flows in the plan.

4. **memchr the BigQuery CSV/TSV text lane.** The one text-parse hot loop in the engine (tsv_to_csv, csv_quote_into, unescape_into) scans byte-at-a-time via `slice::split` closures that don't autovectorize. memchr is already in Cargo.lock transitively — promote it to a direct dep and swap the scans. Small effort, self-contained, only touches the MySQL→BQ CSV lane.

5. **Bigger structural bets, only if a 10M profile justifies them:** the adaptive cgroup-memory backpressure controller (the single technique that hits *both* the memory ceiling and small-core throughput, but large/risky), and the raw MySQL binary-protocol row reader (the biggest structural win on the MySQL leg, but large and correctness-critical — sqlx doesn't cleanly expose the packets). These are real, but they are not cheap. Don't start them before banking #1–4.

**One lever at a time, checksum-gated, measured at 10M and on the 256MB/0.5-core box** — as always.

## 3. What does NOT transfer — and why

QuestDB assumes **owned, persisted, memory-mapped columnar storage on a long-lived server**. apitap owns nothing: it reads a wire format and writes a *different* wire format, once, with no persisted state. That kills these outright:

- **mmap / MemoryCMOR sub-range zero-copy handoff (FD caching).** Explicitly a *same-host* trick that never crosses a socket. apitap's network analog — write the final wire format once into a reusable `out` buffer — is already the correct design. The *only* thing worth salvaging from this item is the async-consumer *shape*, which is exactly what exposes the inline-gzip defect (lever #1). Nothing else.
- **Hardlinked base dictionary files + SymbolMapDiff shipped in WAL.** Requires a *persisted destination dictionary* to hardlink against and diff. apitap has none and never will — it's stateless per run. Only the *local* dict-encode idea survives, and Parquet already does it natively (lever #12).
- **Umbra / German-string 16-byte VARCHAR layout.** The prefix-filter / inlined-equality layout is a *read/query* optimization for a write-once stream — useless. Only its underlying win ("short strings pay zero allocation") transfers, and narrowly: the UUID stack-buffer fix (lever #7).
- **std::simd (portable_simd) fused null-scan / min-max / bloom.** Two reasons it's a non-starter: (a) `std::simd` needs **nightly**, which would break the stable PGO release wheel — mandatory blocker; and (b) apitap streams **row-major**, so values are interleaved with framing and never form a contiguous column to sweep. The only contiguous arrays (Parquet ColBuf) feed stats apitap doesn't currently compute. If we ever compute column stats, use `wide`/`pulp`, not `std::simd`.
- **Symbol columns dictionary-encoded to 4-byte ints on ingest.** On the Parquet leg this is *already on by default* — the task is "verify it isn't disabled," not "build it." On the ClickHouse leg the RowBinary wire still carries full strings, so there's zero wire/CPU benefit; it's a destination-DDL storage choice only.
- **Off-heap `Unsafe.getByte` on a malloc'd recv buffer to escape the JVM heap/GC.** apitap is Rust/tokio — there is no GC and the recv already lands in a `&mut [u8]` with a single kernel→user copy. The "escape the heap" half is a JVM problem we don't have. Only the "no bounds checks" sliver *might* transfer to the transcode loop (lever #10), and only if a profile proves the checks aren't already hoisted by the optimizer.
- **Adaptive recv-buffer sizing across thousands of idle sockets.** QuestDB sizes buffers across many mostly-idle connections. apitap has a fixed small N of always-busy pipes — the idle-sizing logic is pointless. Only "reuse, don't realloc" transfers (lever #8).

Two items are **already banked** — list them so nobody re-implements them: **parse-in-place** (rowbinary.rs push() fast path transcodes straight out of sqlx's zero-copy `Bytes` view) and **straddle-only compaction** (only the partial trailing tuple is buffered across chunk boundaries; the >1MB drain is gated and rare). apitap arrived at both QuestDB designs independently. Nothing to do but confirm.

## 4. Top 3 — what to change and how to measure

### Lever 1 — Move BQ gzip off the async worker thread

**Change.** In `BqLoader::send` (bigquery.rs, the `self.gz.write_all(&self.ndjson)` at :1141), the CPU-bound gzip currently runs inline on the tokio worker, so the socket read and the HTTP upload cannot overlap with compression. Move the compress into `spawn_blocking` (or a dedicated compressor task fed by a **bounded** channel of 1–2 buffers), mirroring the A1 spawn_blocking fix already applied to the Snowflake PUT. The producer hands an owned uncompressed buffer to the blocking compressor; the compressed output flows to the uploader. The in-flight compress queue **must** stay bounded at 1–2 buffers or the 256 MB ceiling is breached — this is the single hard constraint. On a true 0.5-vCPU box there is no spare core, so the win narrows to overlap-with-network (compress while the previous chunk uploads); on ≥1-core tiers it's genuine parallelism.

**Measure.** Run a MySQL→BQ (CSV/gzip lane) transfer at 10M rows, checksum-gated, on the 256MB/0.5-core container. Compare wall time before/after and watch peak RSS to confirm the bounded queue held the ceiling (must not exceed `parallel × chunk_bytes` + 1–2 compress buffers). Then repeat on a ≥1-core tier to confirm the larger overlap win materializes. Confirm the destination row checksum still matches — this is a scheduling change, so bytes must be identical.

### Lever 2 — Parquet DATE stored as `Vec<i32>`

**Change.** `ColBuf::Date` is `Date(Vec<i64>)` (bqparquet.rs:87) and every `flush_row_group` allocates a throwaway `let narrowed: Vec<i32> = v.iter().map(|&d| d as i32).collect()` (:511) purely to feed `write_batch::<Int32Type>`. Store it as `Vec<i32>` from the start: change the variant to `Date(Vec<i32>)`, push `(days + PG_EPOCH_DAYS) as i32` in `push_pg` (:176) keeping the existing infinity guard clamped to `i32::MAX/MIN`, and hand the slice straight to `write_batch` with no narrowing. Update the `len()`/`bytes()`/`clear()` match arms (:124/:139/:151 — split the shared `Date | Ts` arm since Ts stays i64) so `bytes()` reports 4 B/cell, which also tightens `group_bytes()` accounting. This removes one O(rows) alloc+copy per flush and halves the DATE buffer.

**Measure.** DATE roundtrip unit tests already cover this — run them plus a 10M PG→BQ transfer on a date-heavy table, checksum-gated. Confirm identical destination values (esp. the ±infinity edge rows through the clamp) and watch peak RSS drop by ~4 B/cell × rows-per-group × pipes. Wall-time gain is small; the point is the memory profile and removing a per-flush allocation — verify no regression.

### Lever 3 — `Repetition::REQUIRED` + skip def-levels for NOT NULL columns

**Change.** `parquet_field` hardcodes `Repetition::OPTIONAL` for every column (bqparquet.rs:231) and the encoder unconditionally pushes to `self.defs[i]` (0 for null at :461, 1 for present at :467), then passes `Some(defs)` to every `write_batch`. Drive `Repetition` from the source column's nullability (already known in the plan — it's the same `nullable` bool that flows into rowbinary/mysql): NOT NULL → `Repetition::REQUIRED`. For required columns, stop building `self.defs[i]` entirely and pass `None` as the def-level slice to `write_batch`. This drops the whole `Vec<i16>` def array (one i16/row/required-column) from the row-group working set, removes a push from the per-cell framing loop, and yields smaller parquet (no def-level pages → less to gzip and upload). Wide all-NOT-NULL fact tables benefit most. Correctness: if a column declared NOT NULL actually yields a NULL, the writer errors — that's an acceptable loud-fail, matching apitap's existing "NULL in non-nullable" guard at rowbinary.rs:206.

**Measure.** 10M PG→BQ on a wide all-NOT-NULL fact table, checksum-gated, on the 256MB box. Confirm destination checksum matches and peak RSS drops by ~2 B/row × (# required columns) × rows-per-group × pipes. Add a targeted test: a column declared NOT NULL that receives a NULL must produce the loud writer error, not silent corruption. Compare gzipped upload byte count before/after to confirm the def-level pages actually disappeared.

## Completeness gaps (what the 8-subsystem study missed)

# Completeness critique — QuestDB ingestion techniques the 8-subsystem study likely MISSED

The 8 subsystems you covered are heavy on the *in-memory hot loop* (SIMD scans, zero-copy parse, buffer pools, German strings, symbol dicts, backpressure). What they under-cover is the **OS/syscall layer**, **hardware-accelerated hashing/formatting**, and **columnar encoding on the write side**. Those are exactly where apitap's socket-read, gzip, and BigQuery legs live, plus the page-cache angle that bears on the 256 MB ceiling.

Prioritized gaps (highest apitap leverage first):

## Tier 1 — read these next

1. **Parquet dictionary + DELTA_BINARY_PACKED + RLE encoding (write-encode leg, BQ sink)**
   - QuestDB: `core/rust/qdbr/src/parquet_write/` — grep `Encoding::`, `RLE_DICTIONARY`, `DELTA_BINARY_PACKED`, `column_chunk`. This is Rust, directly liftable.
   - apitap: `bqparquet.rs` currently emits typed column builders (raw PLAIN). Adding dictionary encoding for low-cardinality strings and delta/FOR for sorted-ish integer PKs shrinks the Parquet payload → less to gzip and less to upload. Hits leg (b) *and* the BQ load time. Highest single win because it's the same language and you already own the column builders.

2. **Hardware CRC32C for the checksum gate (all legs)**
   - QuestDB: `core/src/main/java/io/questdb/std/Hash.java`, `Vect.java`; Rust hashing in `core/rust/qdbr` — grep `crc32`.
   - apitap: you're "checksum-gated." If that checksum is a general hash/md5 over the byte stream, switch to hardware `crc32c` (`_mm_crc32_u64` on x86-64 SSE4.2, `__crc32d` on aarch64) via the `crc32fast`/`crc32c` crates — multi-GB/s per core, near-free on the small cores. Verify what algorithm the gate uses before assuming a win.

3. **Compression choice: zstd/snappy-in-Parquet or ISA-L deflate instead of gzip (leg b)**
   - QuestDB uses zstd throughout (grep `zstd` / `Zip` in `std`).
   - apitap: gzip on a 0.5 vCPU is your named CPU sink. For BQ, internal Parquet compression (snappy or zstd-1) is far cheaper than gzip-CSV and BQ loads it natively. For the CSV/TSV paths, `zlib-ng`/ISA-L-style SIMD deflate beats stock. Cheap, measurable, one-lever.

## Tier 2 — worth a targeted read

4. **`madvise`/`posix_fadvise(DONTNEED/SEQUENTIAL)` — page-cache pressure vs the 256 MB ceiling**
   - QuestDB: `io/questdb/std/Files.java` (`madvise`, `fadvise`), C in `core/src/main/c/share/files.c` — grep `posix_madvise`, `posix_fadvise`, `MADV_`.
   - apitap: relevant **only if any leg touches disk** (staged BQ Parquet/CSV, gzip temp). In a cgroup-capped 256 MB container, page cache from those files counts against your RSS budget and can trigger reclaim/OOM. `fadvise(DONTNEED)` after streaming a staged chunk keeps the cache footprint flat. If everything is pure socket→socket in memory, skip this.

5. **Software prefetch (`_mm_prefetch` / `__builtin_prefetch`) in the transcode + MySQL-decode inner loops**
   - QuestDB: `core/src/main/c/share/rosti.cpp` and the `vect*.cpp` agg loops — grep `_mm_prefetch`, `__builtin_prefetch`.
   - apitap: in `rowbinary.rs`/`pgcopy.rs` numeric-decode and string-escape loops, and the MySQL per-cell path, prefetching the next row/chunk offset hides memory latency the small cores can't hide with OoO. `core::intrinsics::prefetch_read_data` / `std::arch::_mm_prefetch`. Cheap experiment, one lever.

6. **Fast SIMD integer/decimal formatting for CSV/TSV emission (leg c, CH-TSV and BQ-CSV paths)**
   - QuestDB: `io/questdb/std/Numbers.java` (fast `append`/itoa, no per-digit division loop).
   - apitap: if you format ints/decimals to text for TSV/CSV, a lookup-table or SIMD itoa (`itoa`/`ryu` crates, or QuestDB's two-digit-table trick) beats `format!`/`Display`. Only matters for the text sinks, not the binary COPY/RowBinary paths.

7. **io_uring for batched socket read/write syscalls (legs a & d)**
   - QuestDB: `core/src/main/c/share/io_uring.c`, `io/questdb/std/IOURingImpl.java` — grep `io_uring_setup`, `io_uring_submit`.
   - apitap: reading PG COPY bytes and pushing to dest is syscall-bound on 0.5 vCPU. io_uring amortizes submit/complete and enables registered buffers (zero-copy recv). Big caveat: sqlx/tokio is epoll-based, so this is a heavy rework (tokio-uring or raw ring) — flag as a *research* item, not a quick lever. High ceiling, high cost.

## Tier 3 — note but likely low ROI at your scale

8. **Thread affinity / core pinning** — QuestDB `io/questdb/mp/WorkerPool.java` + `Os.setCurrentThreadAffinity` (C `sched_setaffinity`). Irrelevant at 0.5 vCPU; only helps the larger worker sizes where range-pipes fight over cores. Pin per-pipe threads to cut cross-core cache migration.

9. **Transparent huge pages for the arena buffers** — QuestDB `Files.madvise` with `MADV_HUGEPAGE`. Your tens-of-MB arenas in memcpy-heavy loops could cut TLB misses; `madvise(MADV_HUGEPAGE)` is cheap, explicit `MAP_HUGETLB` is risky in a 256 MB cgroup. Low-medium.

10. **`copy_file_range`/`sendfile` zero-copy** — `files.c`, grep `copy_file_range`, `sendfile`. Only the staged-file→GCS-upload path, and TLS forces user-space anyway. Low.

11. **`O_DIRECT` for staged writes** — bypasses page cache (helps the 256 MB ceiling like #4). Only if you write temp files, and alignment constraints make it fiddly. Low.

12. **Batch-commit / lag amortization** — QuestDB `TableWriter.java` (`o3MaxLag`, commit batching). apitap almost certainly already amortizes via chunked COPY/insert, so probably a no-op. Verify, then drop.

**Explicitly out of scope / skip:** NUMA-aware allocation (single small node, fractional core — no benefit); mmap-the-source (network source, not a file).

The two I'd fund a follow-up read on first: **#1 (Parquet dict/delta — same-language, owns the whole BQ leg)** and **#3 (drop gzip for zstd/snappy-Parquet — kills your named CPU sink)**. Both are one-lever, measurable at 10M, no architecture change.

## High/medium-fit levers (raw)

A direct MySQL binary-protocol row reader — parse the null bitmap + lenenc/fixed values straight from the packet Bytes into encode_value/encode_pg — deletes the per-row Vec+struct+Arc churn and the 150M per-cell type_info clones at 10M rows. This is the 'raw-MySQL-reader' the memory notes as deferred; QuestDB #2+#3 is exactly its blueprint. (The bounds-check-elision angle is un-Rusty and negligible by itself.)

- **[medium] Per-row entity object pooling — no allocation per measurement** — apitap allocates nothing per ROW (it appends into a reused `out` Vec), but it heap-allocates a fresh Vec::with_capacity(chunk + 64KiB) on EVERY send via mem::replace, and the loader drops it after shipping. On small cores this is malloc/free + first-touch page-faulting of tens of MB per chunk, per pipe. This is QuestDB's recycled-event idea applied to apitap's chunk buffers.
  - where: crates/apitap-core/src/driver.rs (Loader trait: return/recycle buffer) + mysql.rs row_worker + postgres.rs copy_out_worker (the mem::replace sites). | impact: Cuts allocator + page-fault overhead on the write-encode/handoff leg; helps most exactly where apitap is weakest (0.5 vCPU / 256 MB). - **[high] Adaptive memory backpressure (multiplicative-decrease / probabilistic-increase) governing block size and partition parallelism** — apitap's parallelism AND chunk_bytes are chosen STATICALLY once at startup (mem_capped_parallel derives a fixed pipe count from CPU + a fixed ~8x-chunk memory model; tier knobs are static; the '1c thrash 180s' was patched with a static conservative default). Replace the fixed cap with a reactive PressureControl: poll cgroup memory.current vs memory.max and, as usage nears the limit, multiplicatively shrink the per-worker coalesce target (chunk_bytes) and/or park a pipe; grow both back with hysteresis when usage recedes. This is the self-tuning batch size that lets a 0.5vCPU/256MB box avoid thrash while a large tier runs bigger batches and more pipes without a hand-set knob. CRITICAL caveat vs QuestDB: you CANNOT catch a cgroup OOM (it's SIGKILL, not a catchable OutOfMemoryError), so the trigger MUST be proactive on a high-water mark of memory.current — never reactive-on-OOM.
  - where: lib.rs mem_capped_parallel / knobs + a shared atomic read by driver row-workers' coalesce threshold and each connector's send loop | impact: Lowers the memory ceiling / eliminates 1c-thrash + OOM-kill risk on small tiers AND raises throughput on large tiers (bigger fused batches, more pipes) — the one technique that hits both the memory ceiling and small-core throughput goals. | effort: large | risk: cgroup v1 vs v2 path differences; polling cost and oscillation (needs hysteresis + min/max clamps); adds a shared atomic on the hot coalesce check; must degrade gracefully if /sys/fs/cgroup is unreadable (non-container runs).
